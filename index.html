Jan 30th, 2014

Bugfix: 1.0.1

Yesterday I released version 1.0.1, which fixes some bugs I discovered while packaging and moves the config file and default database location to the proper application data directory.

This location is: * C:\Users\$USERNAME\AppData\Local\Braxton Schafer\Pyberry on Windows * /Users/$USERNAME/Library/Application Support/Pyberry on Mac * /home/$USERNAME/.local/share/Pyberry on Linux

Additionally, there are packages available for Windows, Mac, and Linux, with no install necessary! Just download from here and unzip.

Major Release: 1.0

This is it, I’ve done it. I’ve tagged and pushed release 1.0. I’ve done as much testing as I can figure, but there’s surely still bugs. If you find a bug, please report it on GitHub so I can fix it!

Download Pyberry 1.0 here.

Jan 22nd, 2014

OpenLibrary and 0.10

I was discovering a host of problems with the Google Books lookup, partially related to poor/inconsistent documentation. So I made the decision to move to OpenLibrary instead. This necessitated a rewrite of much of the lookup code, and I took it as an opportunity to clean up and Pythonize a lot of the code relating to the lookup.

With this change, I decided to tag release 0.10. Yeah, I’m one of those people. This change is certainly bigger than a bugfix but I’m not yet comfortable releasing 1.0. So after 9 comes 10…

Oct 25th, 2013

Bugfix: 0.9.1

Due to a bug I discovered, I’ve now released 0.9.1, which fixes a potential crash when looking up by title or ISBN.

It was possible, if the book didn’t have an author attribute, for Pyberry to crash. This is fixed in 0.9.1.

The release has been tagged in the git repo/GitHub, but I haven’t packaged the release at this point.

Pending Release: 1.0

Pyberry is actually getting close to a 1.0 release! This means that the features I wanted to see in it before it was released are implemented and it’s tested so that it (hopefully) works well and robustly for all use cases.

The only requirement is Google’s google-api-python-client. After that, you should be able to clone it and test it.

Jun 6th, 2013

Backport Complete!

I’ve just pushed version 0.4 of Pyberry to GitHub.  It’s a biggie!  I’ve backported it from Python 3 to Python 2.7 in order to use Google API modules.  Because of that, I was able to make searching by title work.  I’ve additionally cleaned up the code quite a bit, fixed bugs I found going through it, etc.  Check out the new release here, at GitHub.  Now that I’ve pushed past the mega issue, I’m looking forward to ramping up development.

I set a (probably unrealistic) due date for version 0.5 (basically, where most features should be there so that it’s purposeful) to be June 18, next Friday.  I will work on it, but no guarantees about that milestone.

By the way, I still (and always) need testers!  If something doesn’t work, just open an issue on GitHub and I’ll do my best to fix it.  This is very much a learning experience for me.

May 29th, 2013

Py3k to Python 2

Unfortunately, sometimes to move forward, you have to go backwards.  In Pyberry, I’m in the process of moving all the code to Python 2, for library reasons.  I’ve more than met my match in dealing with the data coming from the Google Books API, but Google provides a library to deal with it that’s Python 2.x only.  So, I’ve decided to backport my code.

More details to come - right now, I’m having issues finding decent documentation on the gdata-python libraries, but it’s still looking to be a lot easier.

Feb 21st, 2013

Looking for Release Specialist

The Pyberry project is looking for a Release Specialist - somebody who knows how to package versions for release cross-platform.  Because Pyberry is written in Python, an interpreted language, it can run on many platforms.  We’d like to at least have packages for Windows, Mac OS X and Linux so that a user on these platforms can simply install/run it without having to worry about downloading Python as well.

If you would be willing to do this or help out in any other way, please let me know!  Feel free to browse the code at SourceForge.  If you’re interested, you can email me at braxton.schafer@users.sf.net
